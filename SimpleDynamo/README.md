<p>Replicated Key-Value Storage</p>
<p>This application is a very simplified version of Dynamo. There are three main pieces that have been implemented:</p>
<pre><code>1. Partitioning
2. Replication
3. Failure handling
</code></pre>
<p>The main goal is to provide both availability and linearizability at the same time.
In other words, the implementation will always perform read and write operations successfully even under failures.
At the same time, a read operation always returns the most recent value.
Partitioning and replication are done exactly the way Dynamo does.</p>
<p>Just like <a href="https://github.com/vivekbhalla/SimpleDHT">Simple DHT</a>, it supports and implements
all storage functionalities i.e.  Create server and client threads, open sockets, and respond to incoming requests.</p>
<p>There is support for insert/query/delete operations. Also, there is support for the @ and * queries.</p>
<p>There are always 5 nodes in the system. Adding/removing nodes from the system is not implemented.</p>
<p>There is support for at most 1 node failure at any given time.</p>
<p>All failures are temporary.</p>
<p>When a node recovers, it copies all the object writes it missed during the failure.
This is done by asking the right nodes and then copy from them.</p>
<p>It supports concurrent read/write operations.</p>
<p>It handles a failure happening at the same time with read/write operations.</p>
<p>Replication is done exactly the same way as Dynamo does.
In other words, a (key, value) pair is replicated over three consecutive partitions, starting from the
partition that the key belongs to.</p>
<p>Unlike Dynamo, there are two things which have not been implemented.</p>
<ul>
<li>Virtual nodes: In this implementation we use physical nodes rather than virtual nodes, i.e., all partitions are static and fixed.</li>
<li>Hinted handoff: This implementation does not implement hinted handoff. This means that when there is a failure, replication takes place only on two nodes.</li>
</ul>
<p>All replicas store the same value for each key. This is âper-keyâ consistency.
There is no consistency guarantee across keys. More formally, per-key linearizability is implemented.</p>
<p>Each content provider instance has a node id derived from its emulator port.
This node id is obtained by applying the hash function (i.e., genHash()) to the emulator port.
For example, the node id of the content provider instance running on emulator-5554 should be,</p>
<pre><code>node_id = genHash(â5554â);
</code></pre>
<p>This is necessary to find the correct position of each node in the Dynamo ring.</p>
<p>Use the python scripts to create, run and set the ports of the AVDâs, by using the following commands</p>
<pre><code>python create_avd.py 5
python run_avd.py 5
python set_redir.py 10000
</code></pre>
<p>The redirection ports for the AVDâs will be-</p>
<pre><code>emulator-5554: â5554â - 11108
emulator-5556: â5556â - 11112
emulator-5558: â5558â - 11116
emulator-5560: â5560â - 11120
emulator-5562: â5562â - 11124
</code></pre>
<p>The app opens one server socket that listens on 10000.</p>
<p>Based on the design of Amazon Dynamo, the following features have been implemented:</p>
<h3><a href="#membership" aria-hidden="true" class="anchor" id="user-content-membership"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Membership</h3>
<p>Just as the original Dynamo, every node can know every other node.
This means that each node knows all other nodes in the system and also knows exactly which partition
belongs to which node; any node can forward a request to the correct node without using a ring-based routing.</p>
<h3><a href="#request-routing" aria-hidden="true" class="anchor" id="user-content-request-routing"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Request routing</h3>
<p>Unlike Chord, each Dynamo node knows all other nodes in the system and also knows exactly which
partition belongs to which node. Under no failures, all requests are directly forwarded to the coordinator,
and the coordinator should be in charge of serving read/write operations.</p>
<h3><a href="#chain-replication" aria-hidden="true" class="anchor" id="user-content-chain-replication"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Chain replication</h3>
<p>This replication strategy provides linearizability. In chain replication, a write operation always comes to
the first partition; then it propagates to the next two partitions in sequence. The last partition returns
the result of the write. A read operation always comes to the last partition and reads the value from
the last partition.</p>
<h3><a href="#failure-handling" aria-hidden="true" class="anchor" id="user-content-failure-handling"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Failure handling</h3>
<p>Handling failures should be done very carefully because there can be many corner cases to consider and cover.
Just as the original Dynamo, each request is used to detect a node failure.
For this purpose, a timeout for a socket read is used; and if a node does not respond within the timeout,
it is considered as a failed node.
When a coordinator for a request fails and it does not respond to the request, its successor is contacted
next for the request.</p>
<p>There is a tester for checking if everything is working properly.</p>
<p><strong>Note:</strong> The python scripts and tester are provided by <a href="http://www.cse.buffalo.edu/people/?u=stevko">Prof. Steve Ko</a> from the University at Buffalo.</p>
